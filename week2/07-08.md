# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

<br>

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 어제 못 끝냈던 API 테스트를 오늘 마무리했다. `WebServer.main()`의 accept 루프를 `acceptConnections()`라는 별도 메서드로 뽑아내서, 정상 종료(`listenSocket.isClosed()`)와 진짜 에러를 구분해 처리하도록 고쳤다. 이걸로 서버를 띄웠다 내렸다 할 수 있는 `TestServer` 헬퍼를 만들고, `AuthApiTest`에서 실제 소켓으로 로그인 성공/실패/로그아웃을 검증하는 테스트 3개를 추가했다. 5단계 과제 구현사항에 대한 PR 문서도 작성해서 정리했다. 이어서 6단계 과제 목표(로그인 상태에 따라 다른 화면 보여주기)로 넘어가서, 접근 제어를 담당하는 `SessionFilter`를 만들고, `/index.html`의 로그인 여부에 따른 헤더 부분을 `AUTH_HEADER_START/END` 마커로 감싸서 `IndexPageController`가 치환하도록 구현했다. `/mypage`, `/login`, `/registration` 라우트도 `SessionFilter`로 감싸서 등록했다. 구현하다가 발견한 보안 문제 중 XSS(사용자 이름을 그대로 HTML에 꽂아 넣는 문제)는 `HtmlEscaper`를 만들어서, 쿠키 보안 속성 누락 문제는 `Set-Cookie`에 `HttpOnly; SameSite=Lax`를 추가해서 그 자리에서 바로 고쳤다.
- 아직 끝내지 못한 부분: 구현 과정에서 발생한 이슈들을 기록해놓은 known_issues.md에 남아있는 나머지 이슈(요청 줄 길이 제한 없음, 중복 헤더 처리, Content-Length 검증)는 순서까지 정해두고 내일로 미뤘다. 세션에 비밀번호까지 포함된 User 정보를 통째로 저장하는 문제와 Content-Length를 바이트가 아니라 문자 수로 잘못 세는 문제는 7단계 과제에서 DB 연동/이미지 업로드와 같이 처리하기로 하고 이번엔 손대지 않았다.

<br>

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 로그인 여부에 따른 접근 제어를 `PageController` 안에 private 헬퍼(`currentUser`/`requireLoggedIn`/`requireLoggedOut`)로 넣어 완전히 구현하고 테스트까지 통과시켰다가, 이 방식이 마음에 안 들어서 `SessionFilter`라는 별도 클래스로 통째로 갈아엎었다.
- 그렇게 판단한 이유: 직접 짜고 나서 보니, 로그인 여부를 확인하는 로직이 컨트롤러 안에 계속 반복될 구조라는 게 눈에 들어왔다. `Controller`가 함수형 인터페이스라는 걸 다시 떠올리고, 컨트롤러를 감싸는 형태로 앞단에서 필터링을 하면 "로그인해야 볼 수 있음/로그인하면 못 봄" 같은 접근 규칙을 컨트롤러 로직과 완전히 분리할 수 있겠다고 판단했다.
- 고려했지만 선택하지 않은 대안: 지금 필요한 곳이 3군데뿐이니 컨트롤러 내부 헬퍼로도 충분하다는 생각도 했었지만, 이미 구현해보니 앞으로 로그인 여부를 확인해야 하는 페이지가 늘어날 걸 감안하면 지금 정리해두는 게 낫겠다고 판단해, 구현했던 내용을 전부 필터 방식으로 수정했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: `SessionFilter`가 접근 제어를 전부 맡게 되면서, `/mypage`·`/login`·`/registration`은 결국 `StaticFileServer.serve()`를 그대로 쓰면 되는 거라 `PageController`에 남겨둘 이유가 없어졌다. 그래서 진짜 로직이 있는 `indexPage`만 남기고 클래스 이름도 `IndexPageController`로 바꿨다. 더불어, 코드가 훨씬 간단해지고 보기 편해졌다. 한 번의 판단이 클래스 책임 범위와 이름까지 연쇄적으로 정리하게 만든 셈이다.

<br>

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 어제부터 로그인/로그아웃을 검증할 API 테스트를 만들고 싶었는데, `WebServer.main()`이 `while ((connection = listenSocket.accept()) != null)`으로 멈추는 방법이 없는 무한 루프라서 테스트에서 띄운 서버를 다시 끌 수가 없었다.
- 처음 추측한 원인: 서버를 멈추려면 `WebServer` 자체를 상태를 가진 인스턴스로 바꿔서 `start()`/`stop()` 같은 걸 만들어야 하나 싶었다.
- 원인을 확인하기 위해 시도한 방법: `WebServer`가 원래 과제에서 기본으로 제공된 파일이라는 제약을 다시 떠올리고, 인스턴스화했을 때 생기는 문제들을 하나씩 짚어봤다. `main()`을 그대로 못 쓰고 `join()`으로 우회해야 하는 점, accept 중에 나는 예외가 별도 스레드 안에 갇혀 조용히 죽어버릴 수 있는 점, `start()`/`stop()`을 중복 호출하는 경우까지 신경 써야 하는 점이 걸렸다.
- 실제 원인: 정지 기능이 필요한 부분은 accept 루프 하나뿐이었지, `WebServer` 전체를 상태 있는 객체로 바꿀 필요까지는 없었다.
- 해결 방법: while 루프만 `acceptConnections()`라는 메서드로 뽑아내고, `listenSocket.isClosed()`면 조용히 종료하고 그 외 예외는 그대로 `main()`까지 올라가게 나눠서 처리했다. 원본 구조는 그대로 두고, 테스트에서는 소켓만 닫아서 멈출 수 있게 만든 것이다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 기존 코드에 기능을 추가해야 할 때 구조 전체를 바꾸는 방법부터 생각하지 말고, "정말 바뀌어야 하는 최소 범위가 어디인가"부터 따져보기로 했다.

<br>

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: `PageController`가 `IndexPageController`로 정리되고 접근 제어는 `SessionFilter`가 맡게 되면서, 예전에 만들어뒀던 `PageApiTest`라는 테스트 이름이 지금 코드 구조랑 안 맞는다는 걸 스스로 알아챘다. 하나로 뭉쳐 있던 테스트를 `IndexPageApiTest`와 `SessionFilterApiTest`로 다시 나눴다.
- 그 접근이 도움이 된 이유: 클래스 이름과 책임이 바뀌었는데 테스트 이름만 예전 그대로 남아있으면, 나중에 이 테스트가 정확히 뭘 검증하는 건지 이름만 보고는 알기 어려워진다. 코드 구조가 바뀐 시점에 테스트 이름까지 같이 맞춰두니 그 혼란을 미리 없앨 수 있었다.
- 다음에도 반복하고 싶은 점: 클래스 이름이나 책임 범위를 바꿀 때, 관련된 테스트 파일 이름도 그 자리에서 같이 점검하기.

<br>

- 어떤 행동이었는지: 구현하다가 발견한 보안 이슈(XSS, 쿠키 보안 속성 누락, 세션에 비밀번호까지 포함된 User 전체 저장) 세 가지를 발견하자마자 다 고치려 하지 않고, 각각 지금 고칠지 다음 단계에서 고칠지를 따로 판단했다. XSS와 쿠키 속성은 지금 범위에서 바로 고칠 수 있어서 그 자리에서 고치고, 세션에 User 전체를 저장하는 문제는 DB 연동이 들어오는 7단계 과제에서 손대는 게 맞다고 보고 known_issues.md에 순서를 정해 남겨뒀다.
- 그 접근이 도움이 된 이유: 발견한 문제를 무조건 그 자리에서 다 고치려고 하면 지금 하려던 작업 범위를 벗어나기 쉬운데, "지금 고칠 것"과 "나중에 고칠 것"을 구분해두니 오늘 할 일에 집중하면서도 문제를 놓치지 않을 수 있었다.
- 다음에도 반복하고 싶은 점: 구현 중에 새로운 문제를 발견하면 무조건 바로 고치기보다, 지금 범위에서 고칠 문제인지 아닌지부터 먼저 판단하기.

<br>

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: api 테스트 코드를 짤 때, 시간이 부족하고 어렵다는 이유로 직접 시도해보는 단계를 건너뛰고 바로 완성된 코드를 그대로 타이핑한 경우가 있었다.
- 그렇게 된 원인: 처음 보는 조합이라 스스로 짜면 시간이 오래 걸릴 것 같아서, 오히려 익숙하지 않은 부분일수록 더 빨리 넘어가려고 했다.
- 다음에는 다르게 해볼 구체적인 행동: 처음 다뤄보는 조합일수록 시간이 걸리더라도 일단 내 나름대로 한 번 짜보고, 어디서 막히는지 확인한 다음에 그 막힌 부분만 찾아보기로 한다.
