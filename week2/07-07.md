# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

<br>

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: WAS 과제 goal5(로그인/세션)에 대한 구현 계획을 세운 뒤, 계획한 순서대로 진행했다. `HttpSession`/`SessionStorage`(`ConcurrentHashMap` 기반)를 만들고, 쿠키 헤더를 파싱하는 `CookieParser`를 추가해서 `HttpRequest`가 쿠키를 들고 있게 했다. `AuthController`를 만들어 로그인 성공 시 세션을 만들고 `Set-Cookie`로 SessionId를 내려주고, 실패하면 `/login?error=true`로 리다이렉트하게 했다. 로그아웃은 쿠키에서 SessionId를 꺼내 세션을 지우고 `Max-Age=0`으로 쿠키를 만료시키도록 구현했다. `WebServer.main()`이 소켓 생성/accept 루프만 남도록 라우팅 조립 코드를 `RouterConfig`로 빼고, 매 연결마다 새로 만들던 `StaticFileServer`도 하나만 만들어서 공유하도록 리팩터링했다. 로그인/메인 페이지 정적 파일도 폼 제출 방식으로 고쳐서 실제로 브라우저에서 로그인 → 쿠키 저장 → 로그아웃 → 쿠키 삭제까지 눈으로 확인했다. 구현하다 보니 순수 자바와 JVM 지식이 부족하다는 걸 느껴서, 그 부분은 따로 시간 내서 정리했다. JVM 메모리 구조와 GC, JIT, 제네릭의 타입 소거, HashMap 내부 해싱 방식, 동시성 도구들, 예외 처리까지 하나씩 짚었다.
- 아직 끝내지 못한 부분: 소켓을 직접 열어서 검증하는 API 테스트는 아직 못 만들었다. `WebServer.main()`이 멈추는 방법이 없는 무한 루프라 테스트에서 백그라운드로 띄워도 종료가 안 되는 게 이유라, 서버를 멈추는 최소 기능부터 만드는 걸 내일로 넘겼다. 내일은 과제5 검증을 마무리 한 후 PR을 작성하고, 검증 에이전트의 분석 결과를 바탕으로 구현 사항을 수정할 계획이다. 또, 개인적인 학습 측면에서도 Java/JVM 쪽은 동시성 심화와 I/O 모델(BIO/NIO), TCP 연결 부분이 다음으로 넘어갔다.

<br>

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: `WebServer.main()`에 계속 쌓이던 라우팅 조립 코드(Router에 컨트롤러를 등록하는 부분)를 `config.RouterConfig.createRouter()`라는 별도 파일로 분리했다.
- 그렇게 판단한 이유: `WebServer`는 소켓 생성과 accept 루프에만 집중해야 한다고 생각했는데, 로그인 기능이 추가되면서 라우팅 등록 코드가 계속 늘어나는 게 보였다. `Database`나 `HttpHeaderParser`처럼 조립 로직만 담당하는 static 유틸 스타일로 따로 빼두면, 나중에 라우트가 늘어나도 `WebServer` 자체는 안 건드릴 수 있겠다고 판단했다.
- 고려했지만 선택하지 않은 대안: 분리하는 김에 Router 파일 자체도 config 쪽으로 옮기거나, 아예 router 패키지를 새로 만들어서 Router/RouterConfig를 다 몰아넣고 지금의 config 패키지는 없애는 방법도 생각해봤다. 하지만 그렇게 하면 `RequestHandler(webserver) → config.Router → webserver.Controller/세션`처럼 의존 방향이 뒤섞이고, 원래 한 세트로 움직이던 Router/Controller/RequestHandler가 패키지 경계로 갈라져서 실익 없이 복잡도만 올라간다고 보고 접었다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: config는 구체적인 컨트롤러/세션을 조립하는 역할만 맡고, webserver는 요청 처리 메커니즘 전체가 응집된 채로 유지되는 경계가 명확해졌다. 이 리팩토링과 `AuthController` 기능 추가를 서로 다른 커밋으로 나눈 것도, 하나를 되돌려도 다른 하나는 영향받지 않아야 한다는 기준으로 판단한 것이다.

<br>

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 세션/쿠키 구현을 시작하려니 코드를 어디에 어떻게 짜야 할지 감이 전혀 안 잡혔다. 세션 저장 구조를 만든 뒤에도, 헤더 문자열에서 쿠키 값만 골라내는 과정과 로그인 이후 아무것도 안 해도 요청마다 쿠키가 따라오는 이유가 계속 헷갈렸다.
- 처음 추측한 원인: 세션과 쿠키를 "서버가 알아서 관리해주는 것" 정도로만 막연히 알고 있었지, 실제로 요청/응답의 어느 지점에서 어떤 형태로 값이 오가는지는 코드로 그려본 적이 없었다.
- 원인을 확인하기 위해 시도한 방법: 세션 저장 구조(`HttpSession`/`SessionStorage`)부터 순서대로 직접 짜봤다. 쿠키 파싱도 `headers.get("Cookie")`로 헤더 전체에서 Cookie 값만 뽑아내는 단계와, `CookieParser.parse()`로 그 문자열을 `이름=값` 쌍으로 다시 쪼개는 단계를 나눠서 하나씩 손으로 따라가봤다. 그리고 curl로 로그인해서 받은 `Set-Cookie` 값을 다음 요청에 자동으로 안 실어준다는 걸 직접 겪으면서, 그 사이에서 무슨 일이 일어나는 건지 다시 짚어봤다.
- 실제 원인: 서버 쪽 코드는 문자열을 두 단계로 나눠 쪼개는 게 전부였고, "로그인 이후 아무것도 안 해도 요청마다 쿠키가 따라온다"는 부분은 서버 코드가 아니라 브라우저가 하는 일이었다. 브라우저가 `Set-Cookie` 응답을 받으면 내부 저장소에 기억해뒀다가, 같은 도메인/경로로 가는 모든 요청에 자동으로 `Cookie` 헤더를 붙여서 보내주는 것이었다. curl은 이 저장소가 없어서 매번 `-b` 옵션으로 손수 넣어줘야 했던 것도 그래서였다.
- 해결 방법: 세션 저장 구조와 쿠키 파싱을 단계별로 나눠서 이해한 뒤 구현했고, curl로만 확인하면 이 자동 첨부 과정을 못 보니 최종적으로는 브라우저 개발자도구로 로그인 → 쿠키 저장 → 이후 요청에 자동으로 실려가는 것까지 직접 눈으로 확인했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: "서버가 하는 일"과 "클라이언트(브라우저)가 대신 해주는 일"을 먼저 나눠서 구분하고, 그 경계가 안 보이면 curl처럼 자동화가 없는 도구로 먼저 손으로 재현해보기로 했다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 로그인 기능을 구현하고 나서 로그인 성공/실패만 확인하고 끝내지 않고, "로그아웃 했을 때는 어떻게 되는지도 확인해야 하는거 아니야?"라며 로그아웃 후 쿠키가 실제로 지워지는지까지 스스로 검증 범위를 넓혔다. curl로는 확인 안 되는 부분(정적 페이지의 인라인 스크립트 동작)은 curl 결과만으로 만족하지 않고 브라우저 개발자도구(Network/Application 탭)까지 열어서 재확인했다. 여기서 더 나아가 "이렇게 눈으로 확인한 걸로 끝내도 되나, API 테스트나 E2E 테스트로 자동화할 수는 없나"까지 스스로 물어봐서, 지금 있는 테스트가 단위/통합 테스트까지만 커버하고 있고 진짜 소켓을 여는 API 테스트와는 뭐가 다른지 정리했다. 다만 그러려면 계속 떠 있기만 하는 `WebServer.main()`을 멈출 수 있게 먼저 고쳐야 한다는 선행 작업이 필요하다는 것도 함께 확인하고, 무리해서 다 넣기보다 다음으로 넘기기로 했다.
- 그 접근이 도움이 된 이유: "성공 케이스가 됐으니 끝"이 아니라 관련된 흐름(로그아웃, 쿠키 삭제)까지 스스로 검증 범위에 넣고, 나아가 이 수동 확인을 자동화된 테스트로 남길 수 있는지까지 따져보니, 지금 눈으로 확인한 것과 앞으로 자동으로 지켜져야 할 것 사이의 차이가 명확해졌다.
- 다음에도 반복하고 싶은 점: 기능을 눈으로 확인하고 끝내지 말고, "이걸 테스트로 자동화하려면 뭐가 더 필요한가"까지 항상 한 번 더 물어보기.

<br>

- 어떤 행동이었는지: 제네릭을 배울 때 "그냥 임의의 타입을 담기 위한 것 아니냐"는 내 나름의 추측을 먼저 던지고, 그게 `Object`로도 이미 가능하다는 걸 짚어가며 진짜 취지(타입 안전성을 잃지 않으면서 코드를 재사용하는 것)를 다시 정리했다. HashMap이 비트 연산을 쓰는 이유를 배울 때도 "결국 둘 다 비트 연산 아니냐"는 의문이 들어서 끝까지 캐물었고, 그 과정에서 "비트 연산이 빠른 건 코어 개수 덕분이냐"는 잘못된 가설을 세웠다가 싱글코어에서도 ALU가 여러 비트를 동시에 처리한다는 걸 확인하고 바로잡았다. 이렇게 정리한 개념은 바로 내가 만들고 있는 be-was 코드에 대입해봤다 — 스레드마다 로컬 StringBuilder만 쓰면 StringBuffer가 필요 없다는 것, 연결마다 스레드를 무제한으로 만들면 나중에 native thread 생성 실패로 터질 수 있다는 것까지 되짚었다.
- 그 접근이 도움이 된 이유: 설명을 그대로 받아들이지 않고 내 나름의 가설을 먼저 세워서 틀린 지점을 확인하는 방식으로 정리하니, 왜 그렇게 설계됐는지(하위 호환성, 성능 트레이드오프)까지 같이 남았다. 개념을 내 프로젝트 코드에 바로 대입해본 것도 그냥 지식으로 끝나지 않고 실제로 써먹을 수 있는 형태로 남는 데 도움이 됐다.
- 다음에도 반복하고 싶은 점: 새로운 개념을 배우면 설명만 듣고 넘어가지 말고, 내 나름의 가설을 세워서 맞는지 확인해보고, 지금 만들고 있는 코드에 바로 대입해서 되짚어보기.

<br>

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 로그인/로그아웃 기능을 검증할 소켓 기반 API 테스트를 만들려고 했는데, `WebServer.main()`이 멈추는 방법이 없는 무한 루프라서 테스트에서 백그라운드로 띄워도 끝낼 방법이 없었다. 결국 오늘은 이 테스트를 만들지 못하고 다음 날로 넘겼다.
- 그렇게 된 원인: 서버를 처음 설계할 때 "언젠가 테스트에서 서버를 띄웠다 내렸다 해야 할 수도 있다"는 걸 생각하지 못하고, 계속 떠 있는 것만 가정하고 accept 루프를 짰다.
- 다음에는 다르게 해볼 구체적인 행동: 서버나 장시간 실행되는 컴포넌트를 만들 때는, 정상적으로 켜는 것뿐 아니라 "테스트나 운영 중에 안전하게 멈추려면 어떻게 해야 하나"도 처음 설계 단계에서 같이 생각해보기로 한다.
