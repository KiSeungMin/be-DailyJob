# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

<br>

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 어제 이어서 6단계 과제의 남은 이슈들부터 마무리했다. 요청줄/헤더 줄 길이 제한(8192자), 중복 헤더 거부, `Content-Length` 값 검증(1MB 초과 시 예외)을 추가하고 PR 문서까지 정리했다. 이어서 오늘 진짜 핵심이었던 7단계 과제(게시판 기본 기능)로 넘어가서, H2/JDBC 기반 DB 접근 계층을 구축했다. `users`/`posts`/`comments`/`post_image` 스키마를 설계하고 각각의 DAO를 만들었다. 인메모리 `Database`를 쓰던 `UserController`/`AuthController`를 `UserDao`(H2) 기반으로 전환했고, 세션에 User 객체 전체를 저장하던 것도 `userId`만 저장하고 매 요청마다 DB에서 다시 조회하는 방식으로 바꿨다. 서버를 완전히 재시작해도 회원 데이터가 파일 기반 H2에 그대로 남아있는지 직접 확인했다. 그 와중에 CS/JVM 개인 학습도 이어갔는데, 오늘은 스레드 풀·가상 스레드·디자인 패턴·아키텍처 스타일(헥사고날, DDD) 위주로 짚었다.
- 아직 끝내지 못한 부분: 게시글 작성/좋아요/댓글 기능은 설계(plan1.md)만 끝냈고 코드는 아직 하나도 안 짰다. 템플릿 렌더러도 클래스 틀만 논의하고 실제 구현은 다음날로 넘겼다. `PostDao`/`CommentDao`/`PostImageDao`는 테이블 생성까지만 확인했고 실제 CRUD 왕복은 아직 검증 전이다. 6단계 known_issues 중 Content-Length를 문자 수로 착각하는 문제는 여전히 미해결로 남겨뒀다.

<br>

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 게시글/댓글/마이페이지처럼 동적으로 값을 채워야 하는 화면이 늘어나는 걸 보고, 컨트롤러마다 각자 StringBuilder로 HTML을 조립하던 방식을 그대로 늘리지 않기로 했다. `${key}` 형태의 자리를 정규식으로 찾아 값으로 치환하는 범용 템플릿 유틸리티(`TemplateRenderer`)를 만들어서, 모든 컨트롤러가 값을 HTML에 끼워넣을 땐 반드시 이걸 거치도록 하기로 했다.
- 그렇게 판단한 이유: 이미 겪었던 XSS 버그(`IndexPageController`가 사용자 이름을 끼워넣을 때 이스케이프를 딱 한 군데서 빠뜬 것)를 다시 짚어봤다. 그 버그는 여러 컨트롤러가 제각각 문자열을 조합하다가 생긴 문제였는데, 값을 끼워넣는 지점을 유틸리티 하나로 강제하면 이스케이프를 그 안에서만 제대로 해두면 어디서든 자동으로 적용돼서 같은 종류의 버그가 구조적으로 재발할 수 없겠다고 판단했다. Thymeleaf 같은 완성된 템플릿 엔진을 그냥 가져다 쓰는 방법도 있었지만, 그건 Tomcat/Spring 같은 더 큰 프레임워크가 딸려와서 지금 직접 구현하고 있는 HTTP 파싱/라우팅/세션 처리까지 대신 가져가버리는 구조라 이번 과제 취지(밑바닥부터 직접 구현하기)에 안 맞는다고 보고 제외했다.
- 고려했지만 선택하지 않은 대안: 지금 있는 `<!-- AUTH_HEADER_START/END -->` 마커 방식을 그대로 일반화해서 이름 있는 구간 전체를 완성된 HTML 조각으로 통째로 갈아치우는 방법, 그리고 페이지마다 전용 렌더 메서드를 따로 만드는 방법도 검토했다. 둘 다 구현은 더 단순하지만, "값을 끼워넣을 때 이스케이프를 빠뜬다"는 원래 버그의 재발을 막지는 못한다는 점에서 제외했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 앞으로 동적으로 값을 채워야 하는 화면이 더 늘어나도, 전부 같은 유틸리티 하나만 거치면 이스케이프를 깜빡할 걱정이 구조적으로 없어진다. 반복/조건 처리(댓글 여러 개 렌더링, 개수에 따라 버튼 보이기 등)는 유틸리티에 넣지 않고 그대로 컨트롤러의 평범한 코드로 남겨둬서, 유틸리티 자체는 작게 유지하면서도 학습 목표(직접 구현)를 계속 지킬 수 있었다.

<br>

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 세션에 User 전체 대신 `userId`만 저장하도록 시그니처만 먼저 바꾸고, 실제 재조회 로직은 아직 비워둔 채로 전체 테스트를 돌렸는데 정확히 9개가 실패했다.
- 처음 추측한 원인: 시그니처를 바꾸면서 어딘가 호출부를 놓친 게 아닐까 싶었다.
- 원인을 확인하기 위해 시도한 방법: 짐작만 하지 않고 실패한 테스트 목록을 직접 확인했다. 로그인 안 한 상태를 확인하는 테스트는 다 통과했고, 로그인한 상태를 확인하는 테스트만 실패한다는 패턴이 보였다.
- 실제 원인: 로그인 여부를 확인하는 로직이 여전히 세션에서 더는 존재하지 않는 `"user"`라는 키를 찾고 있어서, 항상 빈 값을 반환하고 있었다.
- 해결 방법: 세션에서 `userId`를 꺼내 DAO로 다시 조회하는 방식으로 바꿔서 직접 타이핑했다. 다시 돌려서 9개를 포함한 전체 테스트가 통과하는 걸 확인했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 테스트가 실패하면 원인을 짐작만 하지 말고, 실패하는 테스트들 사이에 공통된 패턴이 있는지부터 목록으로 확인하기로 했다.

<br>

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 템플릿 유틸리티를 직접 만들기로 하기 전에, 이게 지금까지 지켜온 원칙(REST API+JS로 화면을 그리는 방식은 안 쓴다)과 충돌하지 않는지부터 스스로 다시 확인했다. 완성된 템플릿 엔진(Thymeleaf 등)을 그대로 쓰는 게 문제였던 이유가 "화면을 편하게 만드는 것" 자체가 아니라 "더 큰 프레임워크가 HTTP 처리를 대신 가져가는 것"이었다는 걸 다시 짚어보고, 직접 만드는 작은 치환 유틸리티는 오히려 지금까지의 방향과 맞는다는 걸 확인한 뒤에 진행했다.
- 그 접근이 도움이 된 이유: 새로운 구조를 들이기 전에 지금까지의 원칙과 충돌하는지부터 확인하니, 나중에 "이거 우리가 안 쓰기로 한 방식 아니야?"라는 혼란 없이 일관된 방향으로 계속 진행할 수 있었다.
- 다음에도 반복하고 싶은 점: 새로운 구조나 유틸리티를 들이기 전에, 지금까지 지켜온 원칙과 충돌하는 부분이 없는지 한 번 더 확인하기.

<br>

- 어떤 행동이었는지: 스키마를 설계할 때 지금 요구사항에는 게시글당 이미지가 한 장뿐인데도, `post_image`를 `posts` 테이블의 컬럼 하나로 넣지 않고 별도의 1:N 테이블로 분리했다. 실제 화면/로직은 스펙대로 이미지 한 장만 다루게 제한해두고, 나중에 여러 장을 첨부할 수 있게 되더라도 테이블 구조 자체는 바꾸지 않아도 되게 만들어뒀다.
- 그 접근이 도움이 된 이유: "필요한 것만 만든다"는 원칙과 "나중에 이미지가 여러 장 첨부될 수도 있다"는 현실적인 가능성을 동시에 고려해서, 로직은 지금 스펙에 맞게 단순하게 유지하면서도 스키마를 나중에 다시 바꿀 필요는 없게 만들 수 있었다.
- 다음에도 반복하고 싶은 점: 지금은 1:1처럼 보이는 관계라도 나중에 1:N으로 늘어날 가능성이 뚜렷하면, 로직은 단순하게 유지한 채 테이블 구조만 미리 그렇게 잡아두는 것도 고려하기.

<br>

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: DAO를 구현하다가 "나중에 DB를 다른 걸 쓸 수도 있으니 인터페이스로 감싸두면 어떨까"라는 생각이 들었다. 지금 이 프로젝트는 H2/JDBC로 못박혀 있고 실제로 두 번째 구현체를 만들 계획도 없는데, "나중에 바뀔 수도 있으니까"라는 이유만으로 추상화를 하나 더 넣고 싶어진 것이다.
- 그렇게 된 원인: 구현체가 하나뿐이고 교체 계획도 없는 상태에서, 인터페이스가 주는 "안전망" 느낌에 먼저 끌렸던 것 같다. 돌이켜보면 이건 지금까지 계속 지켜온 "필요한 것만 만들고, 미래를 미리 설계하지 않는다"는 원칙과 반대되는 방향이었고, 지금 와서 보니 오버엔지니어링에 가까운 고민이었다.
- 다음에는 다르게 해볼 구체적인 행동: 추상화를 넣고 싶은 마음이 들 때마다 "지금 실제로 두 번째 구현체가 존재하거나 곧 생길 게 확실한가"부터 먼저 확인하고, 그렇지 않다면 나중에 필요할 때 뽑아내는 쪽(비용이 작은 작업이니 그때 하기)으로 미루기로 한다.
