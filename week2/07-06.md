# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 오전 네트워크 수업을 들으면서 애매하게 알거나 헷갈리는 개념이 꽤 많다는 걸 느꼈다. 오후에는 먼저 WAS 과제 level4를 커밋 단위로 쭉 진행했다. POST 요청 body를 파싱하도록 `HttpHeaderParser`를 새로 만들고 `HttpRequestParser`에 오버로드를 추가했고, `HttpResponse`가 임의의 헤더를 실어 보낼 수 있게 확장했다. 회원가입을 GET에서 POST 방식으로 바꿔서 성공하면 302로 `index.html`로 리다이렉트하게 만들고, 회원가입 폼도 JS로 쿼리스트링을 조립해서 보내던 방식 대신 `<form method="POST">`로 그냥 제출하도록 고쳤다. 그다음엔 `http` 패키지를 parser/request/response로 나누고, `RequestHandler`가 하던 일 중 요청 읽기와 정적 파일 서빙을 각각 `HttpRequestReader`, `StaticFileServer`로 분리하는 리팩터링도 했다. 안티패턴 점검을 받아서 매직 스트링을 `HttpStatus`/`HttpMethod` enum으로 바꾸고 예외 로깅도 손봤고, 마지막에 헤더 이름 대소문자 처리, body를 여러 번에 나눠 읽는 문제, 정적 파일 서빙이 GET 아닌 메소드에도 200을 주던 버그까지 잡아서 PR 문서까지 써두고 브랜치를 push했다. 또, 오전에 네트워크 수업을 들으면서 아직 내가 모르거나 제대로 알지 못하는 개념들이 많다는 것을 깨달았다. 그래서 과제 구현을 마친 뒤에는 아침에 헷갈렸던 것들을 그냥 넘기지 않고 자리 잡아 정리했다. TCP 3-way/4-way handshake, 블로킹/논블로킹 I/O와 이벤트 루프, 가상 스레드, HTTP 메시지 구조와 버전별 차이, 쿠키/세션/JWT, CORS, TLS와 HTTPS, DNS, 캐싱, 로드밸런싱까지 하나씩 짚어봤다.
- 아직 끝내지 못한 부분: 구현을 하면서 예상되는 위험 요소들을 `known_issues.md`에 적어두었는데, (Content-Length 오버플로우, 콜론 없는 헤더 줄 파싱 시 예외, 중복 Content-Length 헤더 처리)은 이번 범위에서는 일부러 미뤘다. 네트워크 쪽은 CSRF, 웹소켓/SSE, Rate Limiting, NAT/스위치/라우터는 다음에 볼 목록으로만 남겨뒀다.

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 리다이렉트 응답에 `Location` 헤더를 실어 보내야 했는데, `HttpResponse`에 `location` 필드 하나만 추가하는 대신 `Map<String, String>` 형태의 범용 헤더를 받는 생성자를 만들었다.
- 그렇게 판단한 이유: 지금 당장은 Location 하나만 필요하지만, 다음 단계에서 로그인을 구현하면 `Set-Cookie` 헤더도 내려줘야 한다는 걸 이미 알고 있었다. 그때 가서 또 필드를 하나 늘리느니 지금 범용으로 만들어두는 게 낫겠다고 판단했다.
- 고려했지만 선택하지 않은 대안: `location`이라는 이름의 필드만 딱 추가하는 방법도 생각했다. 지금 요구사항만 보면 이쪽이 더 간단하지만, 헤더가 하나 더 늘어날 때마다 필드와 생성자를 계속 늘려야 해서 포기했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 기존 4개 인자짜리 생성자는 지우지 않고 5개 인자 생성자에 위임하는 형태로 남겨서 호출부를 안 건드려도 됐다. 나중에 매직 스트링을 정리하면서 `HttpStatus` enum을 추가할 때도 같은 위임 패턴을 그대로 써먹을 수 있었다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: PR 올리기 전에 한 번 더 점검하는 과정에서, 정적 파일 서빙이 요청 경로만 보고 메소드는 전혀 확인하지 않는다는 지적이 나왔다. 예를 들어 POST /index.html처럼 GET이 아닌 요청을 보내도 파일만 있으면 그대로 200이 나가는 상황이었다.
- 처음 추측한 원인: 지적을 그대로 믿지 않고 `RequestHandler`와 `StaticFileServer` 코드를 직접 다시 열어서 확인부터 했다. 라우터에 등록 안 된 경로는 무조건 정적 파일 서빙으로 넘어가는 흐름이라, 그 넘어가는 지점 어딘가에서 메소드 정보가 빠졌겠다고 짐작했다.
- 원인을 확인하기 위해 시도한 방법: `RequestHandler.java`에서 `staticFileServer.serve(...)`를 호출하는 부분과 `StaticFileServer.serve()`의 시그니처를 나란히 비교해봤다.
- 실제 원인: `StaticFileServer.serve()`가 path만 받고 method는 아예 받지 않아서, 파일이 존재하기만 하면 GET이든 POST든 DELETE든 구분 없이 무조건 200을 응답하고 있었다.
- 해결 방법: `serve(method, path)`로 시그니처를 바꿔서 GET이 아니면 파일 존재 여부와 상관없이 바로 404를 내려주도록 수정했다. 405 대신 404를 쓴 건 앞서 `/user/create`를 GET으로 요청했을 때 실패 처리를 정할 때와 같은 이유였다 — 상태 코드로 자원 존재 여부가 드러나지 않게 하려는 것. 기존 테스트를 `serve("GET", path)` 형태로 고치고, POST 요청이 404로 떨어지는지 확인하는 테스트도 추가했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 요청을 다른 처리 로직으로 넘길 때(라우팅 실패 → 정적 파일 폴백 같은 경우), 넘기는 지점에서 필요한 정보(메소드 등)를 빠짐없이 같이 넘기고 있는지부터 확인하기로 했다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: PR을 올리기 전에 한 번 더 점검하는 과정에서 body를 한 번에 다 못 읽을 수도 있다는 지적이 나왔는데, 코드만 고치고 넘어가지 않고 소켓처럼 body를 여러 번에 나눠서 주는 `ChunkedReader`를 테스트용으로 직접 만들어 재현 테스트를 추가했다. 그리고 고친 코드를 잠깐 원래대로 되돌려서 이 테스트가 실제로 예전 버그를 잡아내는지까지 확인한 다음에 다시 복원했다.
- 그 접근이 도움이 된 이유: 테스트를 추가하는 데서 끝내지 않고 "이 테스트가 진짜 버그를 잡아내는가"까지 확인하니, 나중에 또 비슷한 실수를 해도 테스트가 걸러줄 거라는 확신이 생겼다.
- 다음에도 반복하고 싶은 점: 버그를 고친 뒤에는 테스트를 추가하는 데서 멈추지 않고, 수정 전 코드로 되돌려서 그 테스트가 실제로 실패하는지 한 번 확인하기.

- 어떤 행동이었는지: 오전 수업 때 애매하게 남아있던 개념들을 그날 안에 끝까지 붙잡고 정리했다. HTTP와 HTTPS 차이를 "secure가 붙었다" 정도로만 알고 넘어가면 면접에서 바로 되물음당할 것 같아서, TLS가 실제로 보장하는 세 가지(기밀성/무결성/인증)와 그 대가로 핸드셰이크 왕복이 추가돼 초기 연결이 느려진다는 트레이드오프까지 구체적으로 짚었다. 소켓 스트림에서 HTTP 요청이 실제로 어떻게 처리되는지도 궁금해서, 커널이 바이트 스트림을 순서대로 넘겨주는 것과 그 스트림에서 요청 하나의 경계(`\r\n\r\n`, Content-Length)를 찾는 건 완전히 다른 계층의 책임이라는 걸 확인했고, "한 번의 read가 곧 요청 하나가 아니다"라는 함정도 짚으면서 내가 만들고 있는 `HttpRequestParser`, `RequestHandler`가 정확히 이 경계선 중 어디를 책임지는 코드인지 다시 정리했다.
- 그 접근이 도움이 된 이유: 이해가 안 되는 지점을 그냥 넘기지 않고 구체적인 시나리오(카페 공용 와이파이에서의 스니핑 등)와 실제 내 코드(`HttpRequestParser`)에 대입해서 확인하니, 단순히 설명을 들은 것보다 훨씬 오래 남았다. 특히 요청 흐름에서 커널과 응용 계층의 책임을 나눠본 게, 지금 짜고 있는 파서 코드가 왜 이런 구조여야 하는지를 스스로 납득하는 데 도움이 됐다.
- 다음에도 반복하고 싶은 점: 헷갈리는 개념이 나오면 정의만 외우고 넘어가지 말고, 실제 시나리오나 내가 짜고 있는 코드에 대입해보면서 어디까지가 어느 계층의 책임인지 끝까지 따져보기.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 헤더 이름 대소문자 처리를 설계할 때, 당장 브라우저가 `Content-Length`처럼 일관된 표기로 보내니까 대소문자를 구분해도 된다고 판단하고 그 동작을 테스트로까지 남겨뒀다. 그런데 나중에 다시 점검하다가 HTTP 스펙(RFC 7230)에서는 헤더 이름이 대소문자를 구분하면 안 된다는 걸 뒤늦게 확인했다.
- 그렇게 된 원인: 지금 눈에 보이는 브라우저 동작만 근거로 판단하고, 프록시나 다른 클라이언트가 헤더 표기를 다르게 보낼 수 있다는 가능성은 따로 찾아보지 않았다.
- 다음에는 다르게 해볼 구체적인 행동: HTTP처럼 스펙이 명확히 정해져 있는 부분은 실제 동작만 보고 짐작하지 말고, 판단하기 전에 스펙 문서를 먼저 찾아보고 결정하기로 한다.
