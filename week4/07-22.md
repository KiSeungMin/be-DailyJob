# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

<br>

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 어제 절반만 끝냈던 기능 명세서를 오늘 전부 완성했고, 와이어프레임도 거의 다 끝냈습니다. 와이어프레임은 팀원들이 피그마에 남겨준 코멘트만 오늘 반영하면 마무리됩니다.
- 아직 끝내지 못한 부분: 두 개를 다 끝내는 게 목표였는데, 기능 명세서는 완전히 끝냈고 와이어프레임은 피그마 코멘트 반영이 조금 남았습니다. 이건 오늘 자기 전에 반영할 예정이라 내일 일정에는 넘기지 않을 생각입니다.

<br>

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 프론트엔드 특강에서 소개된 FigmaMake와 Orval을 직접 학습해보고, 우리 팀 프로젝트에도 적용하기로 정했습니다. 단순히 "쓴다"에서 그치지 않고 우리 프로젝트에서 어느 단계에 어떻게 끼워 쓰면 좋을지까지 같이 고민했습니다.
- 그렇게 판단한 이유: 특강에서 듣기만 하고 넘어가면 실제로 쓸 수 있는 도구가 되지 못한다고 봤습니다. 마침 지금 우리가 와이어프레임과 API 명세를 다루는 단계라, FigmaMake로 화면 틀을 잡고 Orval로 명세 기반 API 코드를 생성하는 흐름이 우리 작업 순서와 잘 맞겠다고 판단했습니다.
- 고려했지만 선택하지 않은 대안: 특강 내용은 참고만 하고 지금 하던 방식 그대로 진행하는 것도 가능했지만, 그러면 배운 걸 실제로 검증해볼 기회를 놓치는 거라 아깝다고 봤습니다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 아직 본 구현 단계는 아니지만, Orval처럼 명세에서 API 코드를 생성하는 방식을 앞에서 정해두면 나중에 프런트-백 사이 명세 불일치를 줄일 수 있을 거라고 봤습니다. 도구를 프로젝트 흐름에 맞춰 어디에 쓸지 미리 정리해둔 것 자체가 이후 작업 기준이 됐습니다.

<br>

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 경매방 화면을 구성하면서, 알림 기능을 따로 도입할지 아니면 모든 이벤트를 화면에서 다 보여주되 알림은 안 뜨게 할지를 두고 팀에서 1시간 정도 토론이 이어졌습니다.
- 처음 추측한 원인: 둘 중 하나로 금방 정해질 문제라고 생각했습니다.
- 원인을 확인하기 위해 시도한 방법: 각자 어느 쪽을 원하는지, 그리고 그 방식에서 아쉬운 점이나 꼭 들어가야 한다고 보는 부분이 뭔지를 구체적으로 꺼내놓고 비교했습니다.
- 실제 원인: 알림이 경매방 사용 경험에 직접 영향을 주는 부분이라, 각자 중요하게 생각하는 사용자 경험이 달라서 쉽게 한쪽으로 정해지지 않았던 거였습니다.
- 해결 방법: 각자 아쉬운 부분과 꼭 필요한 부분을 잘 제시해준 덕분에, 양쪽 의견을 합친 절충안으로 잘 정리됐습니다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 사용자 경험이 걸린 결정은 각자 중요하게 보는 지점이 다를 수 있으니, 토론 전에 "이 방식에서 꼭 지켜야 할 것 / 포기 못 할 것"부터 먼저 꺼내놓고 시작하기로 했습니다.

<br>

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 오늘 예상했던 계획을 거의 다 끝냈습니다. 시간이 부족해서 예정보다 1시간 정도 더 남아서 마무리하긴 했지만, 그래도 오늘 할 일을 내일로 크게 넘기지 않았습니다.
- 그 접근이 도움이 된 이유: 오늘 몫을 오늘 끝내둔 덕분에 내일 일정에 차질이 생기지 않게 됐습니다. 어제부터 한 논의에 오래 매몰되지 않으려고 신경 쓴 게 이틀째 진도 유지로 이어진 것 같습니다.
- 다음에도 반복하고 싶은 점: 그날 계획한 몫은 조금 더 걸리더라도 되도록 그날 안에 마무리해서 다음 날로 밀리지 않게 하는 흐름을 유지하고 싶습니다.

<br>

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 두 가지가 아쉬웠습니다. 하나는 토큰 관리를 제대로 못 해서 와이어프레임 작업을 원활하게 이어가지 못한 점이고, 다른 하나는 경매방 화면 논의를 1시간이나 끌면서 결단을 빨리 내리지 못한 점입니다.
- 그렇게 된 원인: 토큰은 작업 중에 얼마나 쓰고 있는지를 미리 신경 쓰지 않고 진행하다가 중간에 걸렸고, 경매방 논의는 절충안이 나올 때까지 계속 이야기를 이어가느라 결론을 정하는 시점을 따로 끊지 못했습니다.
- 다음에는 다르게 해볼 구체적인 행동: 도구를 쓸 때는 토큰이 얼마나 남았는지 미리 확인하고 작업 단위를 쪼개서 진행하고, 논의는 일정 시간이 지나면 그때까지 나온 의견으로 제가 먼저 결론을 제안해서 마무리 시점을 만들기로 하겠습니다.
