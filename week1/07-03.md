# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: goal3(GET 회원가입) 작업을 시작하기 전에, `index.html`에서 "회원가입" 링크를 눌러도 이동이 안 되던 버그부터 고쳤다. 원인은 `/registration`처럼 디렉터리를 가리키는 경로를 그냥 파일 경로로 취급해서 읽으려던 거였고, 디렉터리면 `index.html`을 대신 찾도록 고쳤다. 이어서 쿼리 파라미터 파싱(`QueryStringParser` 신설, `HttpRequestParser`/`HttpRequest` 확장)도 구현하고 테스트까지 다 통과하는 걸 확인했다. 두 작업 모두 커밋까지 끝냈다. goal3 전체 구현 순서를 커밋 단위로 정리한 계획 문서(`plan.md`)도 만들었다.
- 아직 끝내지 못한 부분: 라우팅(`Router`/`Controller`/`UserController`)과 회원가입 폼 프론트엔드 연결은 설계만 논의했고 코드는 아직 하나도 안 짰다.

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 쿼리스트링을 파싱하는 로직을 `HttpRequestParser` 안에 그냥 넣지 않고 `QueryStringParser`라는 별도 클래스로 뺐다.
- 그렇게 판단한 이유: POST 방식 폼이 body에 담아 보내는 데이터(`application/x-www-form-urlencoded`)가 쿼리스트링이랑 파싱 규칙이 완전히 똑같다는 걸 확인했다. 지금 정적 리소스 폴더에 로그인/마이페이지/게시글/댓글 화면이 이미 준비돼 있는 걸 보면 조만간 POST 폼도 필요해질 것 같아서, 그때 이 클래스를 그대로 재사용할 수 있게 미리 분리해뒀다.
- 고려했지만 선택하지 않은 대안: `HttpRequestParser.parse()` 메소드 안에서 바로 쪼개서 처리하는 방법. 지금 당장은 코드가 더 짧지만, 나중에 body 파싱이 필요해지면 로직을 복붙하거나 메소드를 다시 쪼개야 해서 포기했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: `QueryStringParser`를 요청 줄 형식 없이 독립적으로 테스트할 수 있게 됐고, `HttpRequestParser`는 "요청이 어떻게 조립되는지"만 보여주는 역할로 단순해졌다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 디렉터리 요청 처리를 위해 `readStaticFile`을 `resolveStaticTarget`으로 리팩터링하고 나서, 직접 코드를 다 고쳤다고 생각했다.
- 처음 추측한 원인: 딱히 이상하다고 느끼지 못했다. 코드 리뷰를 받아보기 전까지는 문제가 있는지조차 몰랐다.
- 원인을 확인하기 위해 시도한 방법: 리뷰를 받아서 바뀐 코드를 한 줄씩 원래 의도와 비교해봤다.
- 실제 원인: 두 가지였다. 정적 루트 경로 문자열에 `static`을 `statc`로 오타 내서 모든 정적 파일 요청이 다 404로 떨어지고 있었고, `buildResponse`에서 content-type을 판단하는 부분이 여전히 원래 요청 경로를 참조하고 있어서 애초에 고치려던 버그(디렉터리 요청 시 content-type이 잘못 나가는 것)가 그대로 남아있었다.
- 해결 방법: 오타를 고치고, content-type 판단을 `target.getName()` 기준으로 바꿨다. 그리고 이 두 문제를 다시 놓치지 않도록 디렉터리 요청 케이스에 대한 테스트를 추가하고, 서버를 직접 띄워서 `curl`로 상태코드/본문/헤더까지 눈으로 확인한 다음에 커밋했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 리팩터링해서 메소드를 새로 뽑아낸 뒤에는, 그 메소드가 필요한 모든 호출부에 실제로 다 연결됐는지 처음부터 끝까지 다시 훑어본다. 특히 경로 같은 문자열 리터럴은 오타가 나도 컴파일 에러가 안 나기 때문에 더 의심하고 본다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 라우팅 구조를 정할 때 지금 당장은 라우트가 하나뿐이니 간단한 분기로도 충분한 게 아닌지 스스로 계속 따져봤다. 그러다 정적 리소스 폴더에 로그인/마이페이지/게시글/댓글 화면이 이미 준비돼 있다는 걸 근거로, 앞으로 라우트가 늘어날 걸로 보고 Map 기반 라우팅 테이블로 가기로 정했다. 이 테이블을 여러 연결이 어떻게 공유할지도 static 필드 방식과 생성자로 주입하는 방식의 장단점을 하나씩 비교해보고, 테스트하기 쉬운 생성자 주입 쪽을 택했다.
- 그 접근이 도움이 된 이유: 구조를 정할 때마다 "왜 이렇게 해야 하는지"를 내가 직접 확인한 근거(폴더 구조, 기존 코드 스타일과의 비교)로 뒷받침하면서 결정하니, 나중에 이 판단을 다시 봐도 왜 그렇게 했는지 스스로 설명할 수 있는 상태로 남았다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 어제 회고에서 "설계 고민에 시간을 너무 많이 써서 실제 구현까지 못 갔다"고 써놓고, 오늘도 똑같이 반복했다. 라우팅 방식, Router/Controller 이름과 위치, 심지어 스프링이 리플렉션으로 컨트롤러를 호출하는 원리까지 파고드는 데 시간을 많이 썼고, 결국 오늘 하루 동안 실제로 코드로 완성해서 커밋한 건 계획한 커밋 4개 중 2개(정적 파일 서빙 버그 수정, 쿼리 파라미터 파싱)뿐이었다.
- 그렇게 된 원인: 궁금한 게 생기면 "일단 지금 필요한 만큼만 이해하고 넘어가자"가 아니라 끝까지 파고드는 습관이 있는데, 오늘은 그게 실제 구현 시간을 꽤 많이 잡아먹었다.
- 다음에는 다르게 해볼 구체적인 행동: 지난번에 세운 계획(설계 고민에 시간 한도를 두거나 핵심 기능부터 동작시키기)을 실제로 지켰는지 회고 때마다 확인하는 절차를 넣는다. 특히 "왜 이렇게 동작하는지"가 당장 구현에 필요한 결정과 무관한 궁금증이면, 그 자리에서 끝까지 파지 말고 따로 메모해뒀다가 구현이 끝난 뒤에 시간 내서 보기로 한다.
