# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 코드 리뷰에서 지적받은 경로 이탈, 쿼리 문자열 섞임, 404 미처리를 수정하고 테스트를 추가했다. 과제 2를 어떻게 구현해나갈지 고민하고, 첫 단계로 응답 처리를 `HttpResponse`(불변 값 객체) + `HttpResponseWriter`(I/O 분리)로 리팩터링했다. 기존 `HttpRequest` + `HttpRequestParser` 구조와 대칭이 맞도록 설계했고, 테스트와 curl 확인까지 마쳤다. 또한 `ContentType` enum 구현과 실제 확장자별 Content-Type 연결 및 검증 과정을 수행했다.
- 아직 끝내지 못한 부분: 3단계 과제에 대한 설계 고민 및 구현 과정이 남았다. 

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: `ContentType` 표현 방식으로 Map 대신 enum을 선택했다.
- 그렇게 판단한 이유: 처음엔 "나중에 확장자가 늘어날 수도 있으니 Map이 더 유연하지 않을까"라고 생각했다. 근데 Map이 유리한 건 런타임에 값이 동적으로 바뀌거나 외부 설정에서 로드할 때인데, 지금은 그게 아니었다. Spring의 `MediaType`이 일반 클래스인 이유가 MIME 타입이 수백 개 넘는 열린 집합이기 때문이라는 걸 확인하고 나니, 6개 고정인 우리 프로젝트는 전제 자체가 달랐다. 그 차이를 이해하고 나서야 enum이 자연스럽게 맞았다.
- 고려했지만 선택하지 않은 대안: Map은 오타를 컴파일 타임에 못 잡고, 일반 클래스는 enum이 공짜로 주는 안전장치를 직접 구현해야 한다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 이 과정에서 실제 프로젝트에 SVG 파일이 7개 있는데 2단계 과제 명세엔 없다는 걸 발견했다. SVG를 enum에 넣기로 한 게 "없는 요구사항 선제 대비"가 아니라 "지금 있는 걸 빠뜨리지 않는 것"이라는 구분도 이때 명확해졌다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: `ContentType`을 Map으로 만들지 enum으로 만들지 결정이 안 됐다. "나중에 확장자가 늘어날 수도 있으니 Map이 더 유연하지 않냐"는 생각이 계속 걸렸다. 3단계 이후 요구사항이 더 있다는 것도 알고 있어서 확장성을 무시하기가 애매했다.
- 처음 추측한 원인: Map이 더 유연한 자료구조라는 건 알고 있으니까, 확장성 면에서는 Map이 당연히 낫다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: Spring이 비슷한 문제를 어떻게 풀었는지 찾아봤다.
- 실제 원인: Spring의 `MediaType`이 일반 클래스인 이유는 MIME 타입이 수백 개 넘는 열린 집합이기 때문이었다. 그래서 외부 설정 파일(`mime.types`)에서 런타임에 동적으로 로드하는 구조를 쓴다. 반면 우리 프로젝트는 지원할 확장자가 6개로 고정돼 있고, "재배포 없이 확장자를 추가해야 한다"는 요구사항도 없다. "확장성이 필요하다"는 직관이 Map이 실제로 유리한 상황(동적 로딩)과 우리 상황을 구분하지 않고 있었던 거다.
- 해결 방법: 고정된 닫힌 집합 → enum. 그 기준이 잡히고 나니 일반 클래스 같은 다른 대안도 자연스럽게 정리됐다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: "확장성이 필요하다"는 생각이 들면, 그 확장이 런타임에 동적으로 일어나야 하는 건지 아니면 그냥 코드 한 줄 추가로 해결되는 건지 먼저 구분하기.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 구현을 시작하기 전에 설계를 먼저 충분히 정리했다. `ContentType`을 뭐로 표현할지, `HttpResponse`를 어떻게 분리할지 먼저 결정하고 나서 코드를 짰다. 경로 이탈 차단과 단순 파일 없음을 둘 다 404로 통일한 것도 같은 맥락이다. 403을 돌려주면 "이 경로가 존재는 하지만 차단됐다"는 정보를 공격자에게 주는 셈이 되기 때문에, 실패 이유를 구분하지 않고 404로 통일해서 아무 정보도 노출하지 않도록 했다.
- 그 접근이 도움이 된 이유: 설계를 먼저 정리하니 구현 도중에 구조를 다시 뜯을 일이 없었다. 404 통일도 먼저 생각하고 결정했기 때문에, 나중에 보안 문제를 뒤늦게 발견하는 상황을 피할 수 있었다.
- 다음에도 반복하고 싶은 점: 구현 전에 "이 결정이 어떤 영향을 미치는지"를 한 번 더 따져보고 시작하기.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 처음 구현할 때 정상적인 요청만 생각했다. `..`가 섞인 경로, 쿼리 문자열이 붙은 URL 같은 예외 케이스는 전혀 고려하지 않았고, 코드 리뷰 지적을 받고 나서야 알았다.
- 그렇게 된 원인: "정상 요청이 잘 동작하면 됐다"고 생각하고 마무리했다. 외부에서 들어오는 값인데도 비정상 입력을 스스로 시도해볼 생각 자체를 안 했다.
- 다음에는 다르게 해볼 구체적인 행동: URL 경로처럼 외부 입력을 다루는 기능을 구현하면, 정상 케이스 확인 후에 `..`, 빈 문자열, 쿼리 문자열 같은 케이스를 바로 직접 한 번씩 넣어보기.
