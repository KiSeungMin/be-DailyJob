# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 요청 라인을 파싱하는 부분(`HttpRequest`, `HttpRequestParser`)을 만들었다. `RequestHandler`에도 붙여서 method/path가 로그에 찍히는 것까지 확인했다. 파서 단위 테스트도 하나 짜서 통과하는 것을 봤다.
- 아직 끝내지 못한 부분: 정작 1단계에서 제일 중요한 "경로에 맞는 html 파일을 읽어서 응답하기"는 아직 손대지 못했다. 지금도 Hello World가 하드코딩된 채로 응답이 나가고 있다.

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 파싱 결과를 배열이나 변수 두 개로 넘기지 않고, `HttpRequest`라는 객체를 따로 만들어서 반환하도록 했다.
- 그렇게 판단한 이유: 지금은 method, path 두 개만 있으면 되지만, 다음 단계로 가면 Content-Type이나 쿼리 파라미터처럼 다뤄야 할 값이 늘어날 것 같았다. 그때마다 반환 타입을 바꾸는 것보다 객체에 필드를 추가하는 쪽이 편할 것 같아서 이렇게 했다.
- 고려했지만 선택하지 않은 대안: `String[]`로 넘기거나 method, path를 각각 인자로 따로 넘기는 방법도 생각은 했다. 지금은 그게 더 간단하지만, 필드가 늘어나면 호출부까지 다 고쳐야 해서 하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: `RequestHandler`는 파싱이 어떻게 되는지 몰라도 되고 getter만 쓰면 된다. 필드가 늘어나도 호출부는 크게 바뀌지 않을 것 같다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: `HttpRequest`에 넣을 필드를 정리하다가, Content-Type도 요청 쪽에 필드로 들고 있어야 하나 싶어서 헷갈렸다.
- 처음 추측한 원인: 클라이언트가 보내는 헤더 중에 Content-Type이 있고, 그걸로 응답 형식을 정하는 줄 알았다.
- 원인을 확인하기 위해 시도한 방법: 요청 헤더와 응답 헤더에 각각 어떤 값이 오가는지 다시 찾아봤다.
- 실제 원인: 요청에 오는 건 Content-Type이 아니라 Accept였다. Content-Type은 요청에 바디가 있을 때 쓰는 헤더고, 응답의 Content-Type은 서버가 확장자나 Accept를 보고 정하는 값이라 애초에 헷갈릴 이유가 없었다.
- 해결 방법: 지금 단계는 확장자만으로 Content-Type을 판별할 수 있어서 헤더까지는 안 봐도 됐다. `HttpRequest`에 관련 필드는 넣지 않기로 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 헤더 이름을 보면 요청 쪽인지 응답 쪽인지부터 구분하고, 헷갈리면 짐작하지 말고 바로 찾아보기로 했다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 파싱 결과를 단순 변수나 배열이 아니라 `HttpRequest`라는 별도 객체로 만들어서, 다음 단계에서 필요해질 값(Content-Type 판별, 쿼리 파라미터 등)을 넣을 자리를 미리 마련해뒀다. 파서와 값 객체를 `webserver`가 아니라 별도 `http` 패키지로 분리한 것도 같은 맥락이다.
- 그 접근이 도움이 된 이유: 지금 당장 필요한 기능(method, path)만 만족시키는 데서 그치지 않고, 이후 자연스럽게 확장할 수 있는 구조를 미리 갖춰둬서 나중에 파서 시그니처나 호출부를 다시 고칠 필요가 줄어들었다.
- 다음에도 반복하고 싶은 점: 당장 필요한 것보다 한 단계 앞을 내다보고 구조를 잡아두는 것.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 위처럼 확장성을 고려해서 설계하는 과정에서, 패키지 위치나 불변 객체 여부, static 여부 같은 세부 사항까지 하나하나 고민하다 보니 정작 오늘 끝내려던 정적 파일 응답 구현까지는 가지 못했다.
- 그렇게 된 원인: 설계 결정 하나하나에 시간을 들이면서, 실제 기능 구현에 쓸 시간이 상대적으로 부족해졌다.
- 다음에는 다르게 해볼 구체적인 행동: 설계 고민에 들이는 시간에 미리 한도를 두거나, 핵심 기능을 먼저 동작하게 만들어 놓고 세부 설계는 그 다음에 다듬는 순서로 진행한다.
