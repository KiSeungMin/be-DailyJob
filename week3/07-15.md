# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 어제 설계해둔 두 번째 PR(이미지 업로드) 구현을 커밋 단위로 쭉 진행했다. 먼저 요청 바디 크기 제한을 1MB에서 30MB로 올리고, `HttpRequestReader`에 바이트 기반으로 읽는 경로를 추가한 뒤 `ConnectionHandler`를 통째로 바이트 기반으로 전환했다. 이 과정에서 한글 닉네임("홍길동김")으로 실제 서버에 회원가입을 걸어봐서, 예전부터 known issue로 남아있던 Content-Length 문자/바이트 착각 버그가 실제로 고쳐졌다는 걸 확인했다. 이어서 `UploadedFile`, 멀티파트 요청을 파싱하는 `MultipartParser`, `HttpRequest.getFile()`을 만들고 `HttpRequestParser`가 Content-Type을 보고 urlencoded/멀티파트로 분기하도록 연결했다(실제 소켓으로 주고받으면서 체크섬까지 맞는지 확인했다). 이미지 매직 바이트로 형식을 검증하는 `ImageFormat` enum과 `images/{연}/{월}/{일}/{UUID}` 형태로 저장하는 `ImageStorage`도 만들었고, 게시글 작성에 이미지 필수 규칙을 적용해서 `PostController`/`PostService`에 연동했다. 정적 파일 서버에 `/images/` 서빙 경로를 추가하고, 메인 화면에 게시글/작성자 이미지를 보여주도록 연결했으며, 게시글 기본 이미지도 설정했다. 이어진 세션에서는 마이페이지 프로필 사진 등록/교체/삭제 로직을 `UserDao`/`UserService`/`UserController`/`mypage.html`까지 전부 연결했고, `GET /` 라우팅 누락도 고쳤다.
- 아직 끝내지 못한 부분: 오전에 직무교육을 듣는 등 다른 일정이 있어서 개발 시작이 늦어졌고, 결국 원래 오늘 안에 끝내려던 두 번째 PR을 다 못 끝냈다. 전체 e2e 검증과 Javadoc 작성, GitHub Pages 게시는 손도 못 댔다. 내일 오전에 Javadoc까지 마저 작성해서 PR을 올릴 계획이다.

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 멀티파트 요청에서 파일 구간의 경계(boundary)를 찾는 방법을, 어제 정해뒀던 "본문 전체를 ISO-8859-1 문자열로 바꿔서 `indexOf`/`split`으로 찾는 방식" 대신 "바이트 배열을 그대로 놓고 경계를 직접 탐색하는 방식"으로 오늘 다시 뒤집었다.
- 그렇게 판단한 이유: 실제 라이브러리(Apache Commons FileUpload, Tomcat)는 이걸 어떻게 처리하는지 찾아보니, 전부 문자열로 바꾸지 않고 바이트를 그대로 스트리밍하며 찾는 방식이었다. 다만 그 라이브러리들이 그렇게 하는 이유(크기를 모르는 스트림을 다뤄야 해서)는 지금 우리 상황(Content-Length로 크기가 이미 정해져 있음)과는 다르다는 것도 같이 확인했다. 그래도 "스트리밍 방식 자체는 나중에 도입하되, 지금 파싱 로직만큼은 실제 구현 방식에 맞춰 바이트로 처리하자"고 판단해서 방식을 바꿨다.
- 고려했지만 선택하지 않은 대안: 어제 정한 대로 문자열 트릭을 그대로 유지하는 방법도 있었다. 이미 검증된 `String` 메소드를 그대로 쓸 수 있어서 구현이 더 간단하고 읽기도 쉬웠지만, 실제 구현 방식과는 거리가 있다는 걸 알게 된 이상 그대로 두고 싶지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: `indexOf` 기반 코드보다 손으로 짠 바이트 탐색 코드는 가독성이 떨어지고 검증된 정도도 낮아졌지만, 실제 구현 방식에 더 가까워졌다는 이득과 맞바꾼 것이다. 이 판단을 내린 이유와 트레이드오프를 plan2.md에 "결정 변경"이라는 절로 따로 남겨서, 나중에 왜 바꿨는지 다시 찾아볼 수 있게 해뒀다.

<br>

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 이미지 업로드/게시글 연동을 다 끝낸 뒤 실제로 업로드한 이미지를 curl로 다운로드해봤는데 404가 났다.
- 처음 추측한 원인: 업로드 자체가 잘못된 게 아닐까 싶었다.
- 원인을 확인하기 위해 시도한 방법: DB와 실제 저장된 파일을 확인해보니 이미지 파일도, DB에 저장된 경로도 다 정상이었다. 그래서 문제를 다운로드(서빙) 쪽으로 좁혀서 `StaticFileHandler` 코드를 다시 봤다.
- 실제 원인: 원인이 두 개나 겹쳐 있었다. plan2.md 설계 단계에서는 `/images/` 경로를 서빙하는 분기를 넣기로 해뒀는데, 정작 실제 구현 체크리스트에는 그 항목이 빠져서 구현 자체가 안 돼 있었다(유닛 테스트는 실제 정적 서버를 안 거치니 이런 건 안 걸러진다). 그 분기를 추가하고 나서도 또 404가 나길래 다시 보니, 저장 경로(`./images` + `/images/2026/...`)를 그대로 이어붙이는 바람에 `./images/images/...`처럼 경로가 겹쳐 있었다. `ImageStorage`는 테스트를 위해 저장 위치를 주입받게 바꿔뒀는데 `StaticFileHandler`는 여전히 경로를 하드코딩하고 있어서, 쓰기와 읽기가 서로 다른 위치를 보고 있었던 것도 그제서야 드러났다.
- 해결 방법: `/images/` 서빙 분기를 추가하고, 경로 접합 로직을 고쳐서 중복 접두사를 제거했다. `StaticFileHandler`도 `ImageStorage`와 같은 방식으로 저장 위치를 주입받게 만들어서 쓰기/읽기가 항상 같은 위치를 보도록 통일했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 설계 문서에는 있는데 실행 체크리스트에는 빠진 항목이 없는지 중간중간 대조해보고, 유닛 테스트만으로는 안 드러나는 부분(정적 파일 서빙처럼 실제 소켓/파일시스템을 거치는 부분)은 반드시 실제 업로드→다운로드까지 눈으로 확인하기로 했다.

<br>

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 컴파일과 단위 테스트가 통과하는 걸로 끝내지 않고, 매 단계마다 실제 서버를 띄워서 회원가입/로그인/이미지 업로드를 curl로 확인했다. 특히 한글 닉네임으로 실제 요청을 보내서 예전부터 남아있던 Content-Length 버그가 진짜로 고쳐졌는지 확인했고, 업로드한 이미지를 다시 다운로드해서 원본과 바이트가 똑같은지(체크섬으로) 확인했다.
- 그 접근이 도움이 된 이유: 컴파일이나 단위 테스트만으로는 안 드러났을 문제(`/images/` 서빙 누락, 경로 중복, 쓰기/읽기 위치 불일치)를 이 과정에서 다 잡아낼 수 있었다.
- 다음에도 반복하고 싶은 점: 파일 업로드/다운로드처럼 바이트가 그대로 오가야 하는 기능은, 단위 테스트가 다 통과해도 반드시 실제로 업로드한 걸 다시 받아보고 원본과 비교하기.

<br>

- 어떤 행동이었는지: 구현하다가 눈에 걸리는 부분을 그냥 넘기지 않았다. `ImageFormat`이 DB에 접근하는 다른 서비스들과 성격이 다른데도 `service` 패키지에 있는 게 이상하다고 짚어서 별도의 `image` 패키지로 옮겼고, `UploadedFile`에 안 쓰이는 필드(`filename`, `contentType`)가 있는 걸 보고 "이걸 왜 남겨두는지"를 따져 물었다. `HttpRequestReader`에서 중복돼 보이는 코드가 있다는 것도 먼저 알아채서 짚었다.
- 그 접근이 도움이 된 이유: 당장 동작에 문제가 없어도 어색한 위치나 안 쓰는 필드를 그때그때 짚어두니, 나중에 "이게 왜 여기 있지"를 다시 고민할 일이 줄었다.
- 다음에도 반복하고 싶은 점: 기능이 다 동작해도 클래스 위치나 안 쓰는 필드처럼 눈에 걸리는 부분이 있으면 그 자리에서 바로 이유를 물어보고 정리하기.

<br>

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 오전에 직무교육을 듣는 등 다른 일정이 있어서 개발 시작이 늦어졌고, 다시 시작한 뒤에도 이미 어제 다 정해뒀던 항목(마이페이지 이미지 삭제 방식)을 처음부터 다시 설명 들으며 재확인하는 데 시간을 꽤 썼다. 그러다 보니 원래 오늘 안에 끝내려던 두 번째 PR을 다 못 끝내고, 전체 e2e 검증과 Javadoc까지 다음 날로 넘어갔다.
- 그렇게 된 원인: 다른 일정으로 텀을 두고 다시 이어서 개발하다 보니, 며칠 전 정리해둔 결정 내용에 대한 감각이 흐려진 상태에서 바로 새 내용을 진행하려 했고, 계획 문서를 먼저 다시 읽기보다 설명을 들으며 그때그때 다시 이해하는 쪽으로 흘러갔다.
- 다음에는 다르게 해볼 구체적인 행동: 다른 일정 때문에 텀을 두고 이어서 개발할 때는, 바로 새 내용을 진행하기 전에 직전에 정리해둔 계획 문서부터 먼저 훑어보고 시작하기로 한다. 내일 오전에는 Javadoc을 마저 작성해서 PR을 올릴 계획이다.
