# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

<br>

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 오늘은 개인 학습과 개발을 각각 진행했다. 네트워크 CS 지식이 부족하다고 느껴서 OSI/TCP-IP 계층, 3/4-way handshake, DNS 조회 과정, HTTP 버전별 차이, 쿠키/세션/JWT/OAuth 2.0·OIDC, HTTPS/TLS, 프록시/로드밸런서/CDN까지 로드맵을 따라 정리했고, 배운 내용을 바로 내가 직접 운영 중인 개인 서버(Nginx + Blue-Green 배포 + Docker MySQL) 구조에 대입해서 점검했다. 개발 쪽에서는 어제 시작한 서비스 레이어 분리를 마저 끝냈다. `AuthController`의 인증 로직을 `AuthService`로 빼고, `PostController.indexPage()`를 여러 private 메서드로 분해한 뒤 `PostService`로, `CommentController`도 `CommentService`로 옮겨서 네 개 컨트롤러 전부 "컨트롤러는 DAO를 모른다"는 원칙으로 정리했다. 예전에 따로따로 만들었던 예외 클래스들도 도메인별 예외 하나(`UserException`/`AuthException`/`CommentException`)와 `Reason` enum으로 통합했다. `RequestHandler`도 `HttpRequestReader`와 이름이 헷갈려서 `ConnectionHandler`로 바꿨고, 로고 링크/로그인 인사말 누락 같은 프런트엔드 버그도 발견해서 고쳤다. 이렇게 첫 번째 PR을 마무리해서 병합했고, 이어서 다음 PR(이미지 업로드)의 설계를 시작했다. HTTP를 바이트 기반으로 다시 다루는 방식, 업로드 파일을 서버가 텍스트/파일로 미리 나눠주는 방식, 이미지를 `static/` 대신 별도 `images/` 폴더에 날짜+UUID로 저장하는 방식, Content-Type 대신 실제 바이트(매직 바이트)로 파일 형식을 검증하는 방식까지 결정해서 계획 문서에 적어뒀다.
- 아직 끝내지 못한 부분: 원래 오늘 두 번째 PR(이미지 업로드)의 설계까지 다 끝내려고 했는데, 결정된 내용을 처음부터 다시 하나하나 제대로 이해하면서 짚어나가는 작업이 세 번째 항목(파일 검증 방식)까지만 끝나고 멈췄다. 나머지 항목(마이페이지 이미지 삭제 처리, `GET /` 라우팅 추가, 회원가입 동시 요청 문제, 멀티파트 파싱 방식, 요청 바디 크기 제한, Javadoc 범위)은 내일 이어서 정리하기로 했고, 실제 코드 구현은 아직 하나도 시작하지 않았다.

<br>

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 회원가입/로그인/게시글/댓글마다 따로 만들었던 예외 클래스 6개를 도메인별로 하나씩(`UserException`, `AuthException`, `CommentException`)으로 합치고, 실패 사유는 `Reason`이라는 enum으로 관리하기로 했다. 처음엔 reason에 따른 메시지를 코드에서 switch문으로 분기했는데, 같은 사유(닉네임 중복)인데도 회원가입과 프로필 수정 두 곳에서 메시지 문자열이 따로 중복되는 걸 발견하고, 최종적으로는 `Reason` enum 상수 자체가 메시지를 필드로 들고 있게 다시 정리했다.
- 그렇게 판단한 이유: 예외 클래스가 계속 늘어나서 폴더가 지저분해지는 게 보였다. 게시글/댓글 기능이 추가되면 예외가 더 늘어날 게 뻔해서, 지금 정리해두면 나중에도 도메인당 파일 하나로 확장 가능한 구조가 될 거라고 판단했다. 메시지를 코드 여기저기 흩어놓지 않고 enum 하나로 모으면, 나중에 문구를 바꾸고 싶을 때 여러 곳을 찾아다니지 않아도 되겠다고 봤다.
- 고려했지만 선택하지 않은 대안: 예외 클래스를 그대로 두고 `service/exception/` 하위 패키지로 옮겨서 폴더만 정리하는 방법도 있었다. 파일 개수가 줄지는 않지만 위험 부담이 없는 방법이었는데, 앞으로 도메인이 늘어날 걸 생각하면 지금 구조를 바꾸는 게 낫다고 보고 접었다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 검증 실패처럼 도메인과 무관한 `ValidationException`은 굳이 enum화하지 않고 자유 문자열로 남겨두고, 도메인마다 분기 처리가 필요한 예외만 `Reason` enum으로 관리하도록 구분했다. 그 결과 예외 관련 파일 개수는 줄고, 메시지의 출처는 enum 하나로 모여서 유지보수가 쉬워졌다.

<br>

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 데이터를 직접 확인해보려고 DataGrip으로 H2에 접속했는데, `Caused by: org.h2.mvstore.MVStoreException: The file is locked: /Users/admin/Documents/softeer/was/be-was/data/bewas.mv.db [2.2.224/7]` 에러가 났다.
- 처음 추측한 원인: WAS 쪽 DB 연결 코드에 뭔가 문제가 있어서 파일이 잠기거나 손상된 게 아닐까, 우리가 만든 코드의 버그일지도 모른다고 의심했다.
- 원인을 확인하기 위해 시도한 방법: WAS 서버를 완전히 종료한 뒤에도 같은 에러가 나는지 확인해보고, H2가 파일 기반 DB(`.mv.db`)를 여는 방식 자체를 다시 찾아봤다.
- 실제 원인: H2의 파일 기반 저장 방식은 기본적으로 한 번에 한 프로세스만 그 파일을 열 수 있게 잠그기 때문에, WAS 서버가 이미 파일을 열어 쓰고 있는 상태에서 DataGrip이 같은 파일에 별도로 접속을 시도하면 잠금 충돌이 나는 것이었다. 우리 코드나 데이터가 잘못된 게 아니라 H2의 정상적인 동작이었다.
- 해결 방법: WAS 서버를 먼저 종료한 뒤에 DataGrip으로 접속해서 데이터를 확인하는 방식으로 우회했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: DB 관련 에러가 나면 우리 코드부터 의심하지 말고, 에러 메시지에 찍힌 라이브러리 이름(여기서는 `org.h2.mvstore`)부터 검색해서 우리 로직 문제인지 그 라이브러리의 원래 동작 방식인지부터 구분해보기로 했다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: `PostController.indexPage()`가 길다는 걸 스스로 짚었는데, 자세히 보니 postId 파싱/최신글 폴백, 게시글이 하나도 없을 때 처리, 도메인 객체를 화면용 데이터로 바꾸는 것, 렌더링까지 네 가지 일을 한 메서드가 다 하고 있었다. 이건 서비스 레이어로 뺄 검증 로직이 있는 게 아니라 순수하게 메서드가 너무 길어서 생긴 문제라는 걸 구분하고, `resolvePost()`/`toPostView()`/`toCommentEntries()`처럼 역할별 private 메서드로 먼저 나눈 뒤 별도 커밋으로 끊었다.
- 그 접근이 도움이 된 이유: "책임이 많다"는 문제를 "서비스 레이어가 필요하다"로 뭉뚱그리지 않고, 실제로 뭐가 문제인지(메서드 분해 문제인지 계층 분리 문제인지)를 구분해서 처리하니 리팩토링 방향이 훨씬 명확해졌다.
- 다음에도 반복하고 싶은 점: 메서드가 길어 보이면 바로 새 클래스나 레이어를 만들기 전에, 그 메서드가 정확히 몇 가지 서로 다른 일을 하고 있는지부터 나열해보기.

<br>

- 어떤 행동이었는지: 서비스 레이어를 분리하다가 "`PostController`가 거의 모든 DAO에 다 접근하고 `PostService`도 `PostDao`에 접근하면, 이게 분리에 의미가 있나?"라는 의문이 들었다. 그래서 "컨트롤러는 DAO를 아예 모른다"는 원칙을 세우고, 사실상 그대로 넘겨주기만 하는 메서드(`resolveAuthorName`, `findPreviousId`, `like`, `exists`)까지 전부 서비스로 옮겨서 네 개 컨트롤러 모두에 일관되게 적용했다. `CommentService`도 게시글 존재 여부를 확인할 때 `PostDao`를 직접 찌르지 않고 `PostService.exists(postId)`를 거치도록 만들어서, 서비스끼리도 서로 다른 도메인의 DAO에 바로 접근하지 않게 했다.
- 그 접근이 도움이 된 이유: 원칙을 어중간하게 적용하지 않고 끝까지 밀어붙이니, "이 컨트롤러는 DAO를 알아도 되고 저 컨트롤러는 안 된다"처럼 예외가 남지 않아서 나중에 코드를 볼 때 규칙이 헷갈릴 일이 없어졌다.
- 다음에도 반복하고 싶은 점: 구조적인 원칙을 세우면 일부만 적용하고 넘어가지 말고, 다소 번거롭더라도 전체에 일관되게 적용하기.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 리팩토링에 너무 집중한 나머지 개발 시간이 예상보다 많이 늘어졌다. 사실 "지금 이 상태로 첫 번째 PR을 올려도 된다"는 결론은 이미 한 번 내렸었는데, 그 이후에도 요청 흐름을 정리한 다이어그램의 색상과 톤을 네 번 넘게 다시 바꾸고, 설명 구조도 여러 번 다시 짰다. 거기다 `ConnectionHandler`/`Router` 관련 구조 논의까지 새로 이어지면서, 실제로 PR을 마무리하고 다음 PR 설계로 넘어가기까지 상당히 오래 걸렸다.
- 그렇게 된 원인: "다듬을수록 좋아진다"는 생각에, 이미 충분하다고 스스로 판단했던 지점을 넘어서도 계속 손을 댔고, 결과물의 완성도를 높이는 일과 원래 목표(PR 마무리)를 구분하지 못했다.
- 다음에는 다르게 해볼 구체적인 행동: "지금 상태로 충분하다"는 결론을 스스로 한 번 내렸으면, 그 이후에 떠오르는 추가 개선 아이디어는 바로 반영하지 말고 일단 원래 목표부터 끝낸 다음에 별도로 처리할지 판단하기로 한다.
