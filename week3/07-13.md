# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

<br>

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 오전 직무교육 시간에 JVM 강의를 들었는데 아는 게 생각보다 얕다는 걸 느껴서, 따로 시간 내서 JVM 구성 요소부터 다시 정리했다. 클래스 로더의 3단계 부모 위임 모델, Loading/Linking/Initialization 각각이 정확히 뭘 대상으로 하는지(인터페이스/enum/익명 클래스까지 포함되고 배열은 제외되는 것), Runtime Data Area가 스레드 공유 여부로 어떻게 나뉘는지, 실무에서 Heap OOM이 왜 나는지까지 이어서 짚었다. 그러다 스레드/동시성(synchronized vs volatile, concurrent 자료구조), 스프링 빈 생명주기와 싱글톤 이유, MSA/Docker/쿠버네티스 오토스케일링, Redis 기반 선착순 대기열 설계까지 궁금한 걸 꼬리 물듯 따라가며 정리했다. 개발 쪽으로는 7단계 과제를 이어가면서, PostController에 몰려있던 HTML 조립 로직(인증 헤더, prev/next 네비게이션, 댓글 목록)을 `template.PostPageRenderer`로 분리하고 `TemplateRenderer.replaceSection` 유틸을 추가했다. 이어서 리팩토링 전에 미뤄뒀던 테스트 커버리지 구멍 3개(댓글 미리보기 경계값 테스트 없음, 500 응답 경로가 한 번도 실행 안 된 테스트, 테스트명과 실제 검증 내용이 다른 `handlesBackslashAndDollarInValues`)를 하나씩 확인하고 고쳤다. 그 다음 컨트롤러들이 검증 로직과 if/try-catch를 너무 많이 떠안고 있다는 생각이 들어서 서비스 레이어 도입을 시작했고, `UserController`부터 `UserService`로 검증/저장 로직을 옮기고 실패 사유별 예외(`UserValidationException`, `DuplicateUserIdException`, `DuplicateNicknameException`, `PasswordMismatchException`)를 나눴다.
- 아직 끝내지 못한 부분: 원래 오늘 7단계의 첫 PR을 올리려고 했는데, 구현하고 보니 코드가 너무 지저분해서 서비스 레이어 분리 작업을 진행하다가 시간이 다 가버렸다. `UserController`만 서비스 레이어 분리가 끝났고, `AuthController`/`PostController`/`CommentController`는 아직 그대로 남아있다. PR1 체크리스트 작성과 실제 PR 오픈은 내일로 넘어갔다.

<br>

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: PR1을 올리기 직전인 지금 시점에 컨트롤러-서비스 레이어 분리를 진행하기로 했다. `UserController`부터 시작해서 하나씩 순서대로 옮기기로 했다.
- 그렇게 판단한 이유: 지금 컨트롤러들을 다시 보니 회원가입 하나에 필수값 누락/4자 미만/아이디 중복/닉네임 중복 같은 검증이 전부 if 체인으로 박혀있고, 같은 종류의 검증 실패인데도 컨트롤러마다 응답 형태(400+평문, redirect, 200 재렌더링)가 제각각이었다. 이게 "나중에 필요할 수도 있으니 미리 대비"하는 상황이 아니라 지금 눈에 보이는 문제라는 게 명확했고, 컨트롤러 개수와 메서드 수가 가장 적은 지금 하는 게 PR2에서 이미지 업로드 기능이 붙어 컨트롤러가 더 늘어난 뒤에 하는 것보다 손댈 범위가 작다고 판단했다.
- 고려했지만 선택하지 않은 대안: PR1은 지금 상태 그대로 먼저 올리고, 서비스 레이어 분리는 완전히 별도 작업이나 PR2 이후로 미루는 방안도 있었다. 과제 요구사항(goal7.md)에는 서비스 레이어를 요구하는 내용이 없어서 이건 순수하게 코드 품질 문제였고, PR1 제출이 하루 늦어지는 걸 감수해야 한다는 것도 알고 있었지만, 그래도 지금 정리하고 넘어가는 쪽을 택했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: `UserController.create()`/`updateProfile()`이 검증 규칙을 직접 들고 있지 않고 `UserService`를 호출한 뒤 예외별로 원하는 응답으로 번역만 하게 되면서 컨트롤러가 눈에 띄게 얇아졌다. 대신 실패 사유마다 예외 클래스를 하나씩 만들다 보니 커밋 단위를 어디까지 나눠야 하는지(예외 클래스들이 신규 파일인데 테스트가 아니라 구현이라는 것)가 헷갈려서, `src/main`이냐 `src/test`냐를 기준으로 다시 정리해야 했다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: `PostPageRenderer` 분리를 끝내고 실제 서버를 띄워 curl로 e2e 검증을 하던 중, 회원가입 요청 하나가 응답 없이 2분 동안 멈춰서 타임아웃이 났다.
- 처음 추측한 원인: 방금 막 끝낸 리팩토링에서 뭔가 깨진 게 아닐까 의심했다.
- 원인을 확인하기 위해 시도한 방법: 서버 프로세스가 살아있는지부터 확인하고, 서버 로그를 tail로 다시 살펴봤다.
- 실제 원인: 새로 생긴 버그가 아니라, 등록 데이터에 한글 이름이 들어가면서 이전부터 알고 있던 Content-Length 관련 known issue(문자 수와 바이트 수를 착각하는 문제)가 그대로 발동한 것이었다. 서버 자체는 살아있었고 등록도 결국 서버 쪽에서는 처리가 됐었다.
- 해결 방법: 한글 대신 ASCII 데이터로 다시 요청을 보내서 우회하고, 나머지 검증 흐름(로그인, 헤더 렌더링, 게시글/댓글/좋아요)은 정상적으로 진행했다. Content-Length 버그 자체는 이번에도 고치지 않고 known issue로 남겨뒀다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: e2e 검증 중 응답이 멈추면 바로 "방금 바꾼 코드가 원인이다"라고 단정하지 말고, 알려진 이슈 목록부터 먼저 확인하기로 했다. 그리고 한글이 섞인 데이터로 검증할 때는 이 known issue가 끼어들 수 있다는 걸 감안해서, 필요하면 ASCII로 먼저 흐름을 확인한 뒤에 한글 케이스를 시도하기로 한다.

<br>

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: `PostController.indexPage()` 하나가 조회 로직 + 인증 헤더 조립 + prev/next 네비게이션 조립 + 댓글 목록 조립을 다 떠안고 있는 걸 보고, 이걸 컨트롤러 안에 그대로 두지 않고 `template` 패키지에 `PostPageRenderer`라는 별도 클래스로 분리했다. DB나 파일 I/O는 전혀 안 하고 이미 조회된 데이터만 받아서 HTML을 조립해 돌려주는 순수 함수 형태로 만들었다.
- 그 접근이 도움이 된 이유: 컨트롤러에서 상수 9개와 HTML 조립용 private 메서드 9개가 통째로 사라지면서 컨트롤러는 파라미터 파싱, DAO 조회, 렌더러 위임, 응답 래핑만 하게 됐다. 순수 함수라서 `PostPageRendererTest`로 분리해 테스트하기도 훨씬 쉬워졌다.
- 다음에도 반복하고 싶은 점: 도메인/페이지마다 무조건 Renderer를 만드는 게 아니라, 단순 치환 1~2개짜리는 컨트롤러에 그냥 둬도 되고 조건부 섹션 교체+하위 리스트 조립처럼 조립 로직 자체가 하나의 독립된 책임이 될 때만 분리한다는 기준을 세워둔 것도 계속 가져가고 싶다.

<br>

- 어떤 행동이었는지: 아침 직무교육에서 JVM 강의를 듣다가 스스로 부족하다고 느낀 지점(클래스 로더 위임 모델, 로딩/링킹/초기화의 정확한 대상, 메모리 영역별로 뭐가 저장되는지)을 그냥 넘기지 않고 그날 바로 붙잡고 정리했다.
- 그 접근이 도움이 된 이유: 수업 때는 "대충 아는 것 같은" 느낌이었던 부분들이, 직접 예시 코드를 놓고 하나씩 따라가며 정리하니까 "static final int 상수는 초기화를 안 트리거한다"처럼 헷갈렸던 디테일까지 명확해졌다.
- 다음에도 반복하고 싶은 점: 수업 중에 애매하게 넘어간 개념은 그날 안에 바로 예시를 만들어서 확인해보는 습관을 계속 가져가기로 한다.

<br>

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 게시글 상세 페이지를 처음 만들 때 설계를 제대로 안 하고 구현부터 들어가서, 인증 헤더/네비게이션/댓글 목록 조립 로직이 컨트롤러 안에 계속 쌓인 뒤에야 `PostPageRenderer`로 다시 빼내는 리팩토링을 거쳐야 했다.
- 그렇게 된 원인: 처음 이 페이지를 만들 때 조립 단계가 앞으로 얼마나 늘어날지 미리 가늠해보지 않고, 일단 되는 대로 컨트롤러 안에 채워 넣었다.
- 다음에는 다르게 해볼 구체적인 행동: 새 페이지나 컨트롤러를 만들기 전에 필요한 조립 단계(조건부 섹션 교체, 하위 리스트 렌더링 등)를 먼저 나열해보고, 그 개수나 복잡도가 일정 수준을 넘을 것 같으면 구현 시작 시점부터 Renderer를 분리해서 시작하기로 한다.
